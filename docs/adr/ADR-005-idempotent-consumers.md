# ADR-005 — Idempotent Inbound Consumers with Event ID Deduplication

**Status:** Accepted  
**Date:** 2026-05-03  
**Scenario:** C — Omnichannel Commerce Core (VerdeMart)

---

## Context

Commerce Core consumes events from RabbitMQ sent by external systems (WMS stock updates, POS sales, shipment status changes). RabbitMQ delivers messages **at-least-once** — under failure conditions (consumer crash mid-processing, connection reset, acknowledgment timeout), the same message may be delivered more than once.

If a "stock decremented by 5" message is processed twice, nopCommerce's stock replica drops by 10. If a "order shipped" event fires twice, the customer may receive two shipping notifications.

**Driver:** QAS-5 requires full stock reconciliation in < 2 min after WMS reconnection with **no duplicate deductions** — even when buffered events are redelivered or replayed.

---

## Decision

All inbound RabbitMQ consumers in Commerce Core are **idempotent**: processing the same message twice produces the same result as processing it once.

Idempotency is achieved via **event ID deduplication**:
1. Every inbound event carries a unique `EventId` (UUID generated by the producer)
2. Commerce Core maintains a `ProcessedEvent` table with `(EventId, ProcessedAt)`
3. Before processing, the consumer checks whether `EventId` is already in the table
4. If already processed, the message is acknowledged and discarded
5. If new, the consumer processes the message and writes the `EventId` to `ProcessedEvent` in the same transaction

---

## Rationale

At-least-once delivery is a fundamental property of message brokers in the face of failures. Designing consumers to require exactly-once delivery puts the responsibility on the broker, which either cannot guarantee it or does so at significant performance cost.

Idempotent consumers are simpler and more robust: they work correctly regardless of how many times a message is delivered. The deduplication table is cheap (a single indexed lookup per message) and provides an audit trail.

The choice of event-level deduplication (by `EventId`) rather than operation-level idempotency (e.g., "set stock to X" rather than "decrement stock by 5") is made because:
- Not all producers can guarantee set-based semantics (legacy stubs may produce delta events)
- Event ID tracking works uniformly across all event types
- The deduplication table is reusable across all consumers

---

## Rejected Alternatives

### Alternative 1: Use exactly-once semantics at the broker level

Configure RabbitMQ with publisher confirms and consumer acknowledgment such that each message is delivered exactly once.

**Rejected because:** RabbitMQ does not provide true exactly-once delivery in the general case (it provides at-least-once). Kafka with idempotent producers and transactions can approach exactly-once, but we chose RabbitMQ (ADR-001). Even with Kafka, exactly-once requires careful configuration and adds performance overhead.

### Alternative 2: Make each operation naturally idempotent (set semantics)

Instead of "decrement stock by 5", producers send "set stock to 45". Processing the message twice sets stock to 45 twice — same result.

**Rejected as the primary strategy** because it requires all surrounding system stubs and future real integrations to use set-based semantics. This is a constraint on producers that we cannot guarantee. Delta events are common in real warehouse and ERP systems. The deduplication approach works regardless of event semantics.

**This approach is still preferred where feasible** — e.g., shipment status updates ("order is now In Transit") are naturally idempotent and do not require deduplication. The deduplication table is the fallback for delta events.

### Alternative 3: Accept duplicate processing and compensate downstream

Allow duplicates and add reconciliation logic to detect and correct the resulting inconsistencies.

**Rejected because:** Compensation logic is more complex than prevention. Detecting that stock was decremented twice requires knowing the expected stock level from an authoritative source — which requires an additional round-trip to the WMS. Prevention via deduplication is strictly simpler.

---

## Consequences

- **Positive:** Consumer behavior is correct under RabbitMQ's at-least-once delivery guarantee
- **Positive:** Deduplication table provides an audit trail of all processed events
- **Positive:** Works uniformly across all event types regardless of producer semantics
- **Negative:** Additional DB write per consumed message (deduplication entry)
- **Negative:** Deduplication table grows over time; requires a retention policy (e.g., keep last 7 days)
- **Mitigation:** Index on `EventId`; background cleanup task for entries older than retention window
